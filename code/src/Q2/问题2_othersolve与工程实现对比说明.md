# 问题2：`othersolve.py` 与工程实现（`solve_q2.py` / `cpsat_q2.py`）详细对比说明

本文档对参考脚本 `othersolve.py` 与当前仓库中问题2工程求解链路进行**逐项对照**，便于论文写作、答辩说明与后续改模时定位差异。**结论先行**：两者在「析取图 + CP-SAT + 设备互斥 + 车间内前驱」上思路相近，但在**数据来源、转运与起点建模、并联台数是否在 CP 内决策、求解后处理与校验口径**上存在可核查的差异；因此 **makespan 数值不必一致**，更不能在未做同一校验的前提下直接断言「谁更优」。

## 3. 工序与展开

### 3.1 参考脚本 `othersolve.py`

- 工序以元组形式写在 `RAW_OPS`：`工序ID、车间字母、设备列表(简称+效率)、工程量、前驱列表`。
- C3–C5 重复三轮已 **手工展开** 为 `C3_1…C5_3` 等 ID，前驱链在数据中写死。

### 3.2 工程实现

- 从 Excel 解析 `RawOperation`，由 `expand_repeated_operations` 按赛题规则展开（含「多工序同重复次数」的交错展开等）。
- 前驱关系由 `_build_predecessors`：**同一车间**内按 `sequence_key`、`op_id` 排序后链式连接。

**一致性前提**：若 Excel 与 `RAW_OPS` 在工程量、效率、重复规则上已人工核对一致，则**展开后的工序条数与链式先后**应与参考脚本一致（你已做过核对时可在此打勾）。

## 4. 作业时长（秒）

### 4.1 参考脚本

- `wt(q, n, v) = ceil(q * 3600 / (n * v))`，其中效率 `v` 为 **m³/h**，与常见赛题口径一致。

### 4.2 工程实现

- `calc_work_duration(quantity, efficiency, unit)`：支持 `m³/h` 等；单类 `n` 台并联时用 **`efficiency * n`** 作为合效率再取整秒。

**注意**：在「单类用满台数」时，两边在公式上等价于「总效率 = 单台效率 × 台数」；若 Excel 中单位字段与脚本隐含不一致，才会出现秒级偏差。

---

## 5. 并联台数与双设备工序

### 5.1 参考脚本（离線）

- `op_dev_durations`：  
  - **单类**：`n = 该类型总台数`，`t = wt(q,n,v)`。  
  - **双类**：对 `n1∈[1,N1]`、`n2∈[1,N2]` 枚举，取 `max(t1,t2)` 最小的组合，得到每类 `(台数, 该类持续秒数)` 与工序阶段时长 `op_dur`。

### 5.2 工程实现

- **`_isolated_best_alloc`**：与上节逻辑对齐，用于 **`fixed_alloc` 分支**（与 `othersolve` 同思路：CP 内只分派机号与顺序，**不再改 n**）。
- **`variable_n` 分支**：在 CP 内用 `AddElement` 等把 `n` 与持续时长联动，可行域 **不小于** 离線固定分支；最优 makespan 可能相同或更优/更差，取决于全局耦合。

---

## 6. 转运与「从何处出发」（重点）

### 6.1 参考脚本

- 距离节点：**基地** 与 **A–E**；`trans(a,b)=ceil(_RAW_DIST/2)`，即 **全局 2 m/s**。
- CP 中转运约束主要出现在：**同一台物理设备上**，两道**真实工序** `p→q` 且 `p` 在 `q` 之前时，`q` 的开工需考虑 `trans(p.workshop, q.workshop)`（同车间为 0 则不加）。
- **未**在模型中显式加入与 `Qbase/validators.validate_transport` 完全同构的「每台设备从 **班组1** 出发再进第一道任务」的**逐机链式**表达；与工程里「班组1 + 距离矩阵 + 每台 speed」的校验口径**是否逐式等价**，需单独论证或做反例检验。

### 6.2 工程实现

- 距离：`parse_distances` → `distance_between`；转运：`calc_transport_time(distance, device.speed)`，**可随设备变化**。
- 每台物理设备（按 `dtype` + 序号 `k`）增加 **虚拟首工序**（零时长），`workshop` 取 **`Device.initial_location`（班组1）**，与真实工序一起参与同一 `(dtype,k)` 上的 `NoOverlap` 与排序弧，从而使 **首段「班组1 → 第一道真工序车间」** 进入与 `validate_transport` 一致的离散表达。

**对比结论（表述建议）**：

- 不能简单等同于「参考脚本不算从班组1 出发」；更准确是：**参考脚本用「基地」与车间间弧表达转运，工程用「班组1 节点 + 虚拟首工序 + Excel」表达首段与跨车间**；两套写法在**数学上是否严格等价**取决于参考脚本是否对「每机第一道任务」补足了与基地/班组1 等价的约束——这是 makespan 与甘特差异的**候选原因之一**，需用**同一校验器**对参考解做验算才能定论。

## 7. CP-SAT 结构对照（简述）

| 构件 | 参考脚本 | 工程实现 |
|------|----------|----------|
| 工序起止 | `op_start`、`op_end`，`op_end = op_start + op_dur`（`op_dur` 离線固定） | 同；`variable_n` 时 `op_dur` 由变量与 `AddElement` 决定 |
| 分配 | `assign[oid][dtype][k]`，`sum == n_need` | 同 |
| 机上占用 | `NewOptionalIntervalVar(start, size, end, is_present)` | 同；固定分支 `size` 为整数常数 |
| 顺序 + 转运 | `p_before_q` + `OnlyEnforceIf` | 同 |
| 虚拟首工序 | 无 | 有（见 §6.2） |

## 8. 目标函数与「甘特图观感」

| 项目 | 参考脚本 | 工程实现 |
|------|----------|----------|
| 第一阶段 | 最小化 makespan | 同 |
| 第二阶段 | 无 | 固定 makespan = 一阶段最优值，最小化 **真实工序 `op_start` 之和**，并对 `op_start` 加 **hint**，减轻多最优解中「开工整体偏晚、中间空档大」的 tie |

说明：在 makespan 已最优的前提下，**压缩开工和**不会改变最优 makespan 值，但通常会改善**时间轴左移与多车间尽早并行**的视觉效果（若仍存在大块空闲，多与**瓶颈设备与工序结构**有关，而非单纯目标函数少写一项）。

## 10. 为何可能出现「参考 makespan 更小」的现象

在**工序参数已核对一致**的前提下，仍可能出现参考 notebook 的 makespan（如文献中常见的约 163989 s）与工程 CP（如约 164315 s）**不一致**，常见原因包括（可多选并存）：

1. **转运与起点建模差异**（§6）：工程侧对「从班组1 出发」的显式建模更贴近当前 `validators`；参考脚本若对首段约束较弱或与「基地」节点映射不完全一致，可能出现**不同可行域**或**不同紧度**。
2. **速度是否全局为 2 m/s**：参考为 `ceil(d/2)`；工程为每台 `speed`。若 Excel 中均为 2 m/s 且关键边与 `_RAW_DIST` 一致，此项影响会缩小，但**结构仍可能不同**（虚拟首工序等）。
3. **并联在 CP 内是否可动**：工程 `variable_n` 分支与离線固定分支全局联立；与仅离線固定后再 CP 的参考脚本**可行域未必相同**。
4. **求解时间上限与最优性**：参考可设 180 s、300 s 等；工程默认 `--time-limit` 分配在固定/变量两链上；**FEASIBLE** 与 **OPTIMAL** 对边界有影响。
5. **数值上界 HORIZON**：参考 `400000`，工程默认 `500000`，一般不影响最优 makespan，极端实例除外。

## 11. 建议的「公平对比」操作流程（写论文/答辩用）

1. **统一输入**：确认 Excel 与 `RAW_OPS` 在工程量、效率、重复、车间链上一致（你已检查工序参数时可引用本节）。  
2. **统一校验**：将参考脚本导出的（设备、起止时间、工序号）转换为与 `ScheduleRecord` 等价结构，调用 **`validate_schedule`**；若出现转运/互斥类报错，则说明参考解在**工程口径**下不可行或需解释建模差异。  
3. **统一时间预算**：对两边设置相近的 CP 时间上限与线程数，再比较 makespan 与甘特。  
4. **记录分支标签**：工程解日志中的 `best_ms=… pick=fixed_alloc::… / variable_n::…` 便于说明最终采用了哪条建模链。
